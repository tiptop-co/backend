# 12 факторов

> В документе описано соответствие проекта принципам `Twelve-Factor-App`

## 1. Codebase

Проект разделен на три репозитория:

1. Репозиторий для фронтенда
2. Репозиторий для бекенда
3. Репозиторий для документации

Каждый репозиторий содержит соответствующую кодовую базу, которая версионируется средствами Git.

## 2. Dependencies

Все зависимости проекта явно описаны и их версии зафиксированы в `go.mod, go.sum` для бекенда и `package.json` для фронтенда соответственно.

Среда выполнения компонент изолирована с помощью применения контейнеризации Docker.

## 3. Config

Исполняемый код полностью отделен от конфигурации.
Конфигурация приложения происходит с помощью переменных окружения.

## 4. Backing services

В качестве внешнего сервиса в проекте используется PostgreSQL.
Данный компонент развертывается отдельно с своем Docker-контейнере.

Основной код приложения не зависит от конкретного компонента.
При смене БД потребуется изменить только драйвер и методы слоя репозитория - методы бизнес-логики изменения не требуют.

## 5. Build, release, run

Процессы сборки и релиза разделены.
На этапе сборки производится сборка соответствующих Docker-контейнеров. Для выполнения данные контейнеры запускаются.

## 6. Processes

Приложение является `Stateless`, что позволяет масштабировать бекенд на множество реплик.
Данные персистентно хранятся в PostgreSQL, а не памяти процесса.

## 7. Port binding

С сетевой точки зрения приложение не зависит от внешних компонент и принимает запросы на определенном порту.
Проброс порта производится при запуске контейнера с приложением.

## 8. Concurrency

Бекенд является `Stateless`, что позволяет масштабировать его на множество реплик.

Для эффективной утилизации ресурсов в рамках одного бекенда используется HTTP-фреймворк, который создает зеленый поток на каждый запрос, что позволяет обрабатывать запросы конкурентно.

## 9. Disposability

Запуск приложения из собранного контейнера происходит быстро.
Для корректного завершения приложения реализован механизм Gracefull Shutdown.

## 10. Dev/prod parity

Код не зависит от окружения.
Окружение конфигурируется с помощью переменных окружения.

## 11. Logs

Приложение пишет логи в стандартный поток вывода и не занимается их интерпритацией/хранением.
Агрегация логов - задача внешнего компонента.

## 12. Admin processes

Скрипты для проведения миграции не являются частью приложения и выполняются в отдельных контейнерах как отдельные разовые процессы.
